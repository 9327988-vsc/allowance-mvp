# MEMORY SYSTEM — 메모리 운영 + 이식 방법

세션 간에 맥락·결정·선호를 잃지 않도록, 어시스턴트는 파일 기반 메모리를 누적한다. 이 문서는 (1) 메모리 운영 방식과 (2) 다른 PC·다른 Claude Code 인스턴스로 이식하는 방법을 정의한다.

---

## 1. 메모리의 종류

| 종류 | 무엇을 담나 | 언제 저장하나 |
|---|---|---|
| **user** | 사용자의 역할·목표·지식 수준·선호 | 사용자에 대한 정보를 알게 됐을 때 |
| **feedback** | 작업 방식에 대한 지침 (피해야 할 것 + 반복할 것) | 사용자가 접근을 교정하거나, 비자명한 접근을 확인해줬을 때 |
| **project** | 진행 중 작업·목표·결정·사고의 맥락 | 누가·무엇을·왜·언제 하는지 알게 됐을 때 |
| **reference** | 외부 시스템(이슈 트래커·대시보드 등)의 위치·용도 | 외부 자원과 그 목적을 알게 됐을 때 |

### 저장하지 않는 것
- 코드를 읽으면 알 수 있는 것 (아키텍처·파일 경로·컨벤션)
- git 이력으로 알 수 있는 것 (변경 이력·누가 바꿨나)
- 디버깅 해법 (코드와 커밋 메시지에 있음)
- CLAUDE.md에 이미 있는 것
- 현재 세션에서만 쓰는 일시적 상태 (그건 과업·계획으로)

---

## 2. 메모리 저장 절차 (2단계)

**1단계** — 메모리를 개별 파일로 작성한다 (예: `user_role.md`, `feedback_testing.md`).
```markdown
---
name: {짧은-kebab-슬러그}
description: {한 줄 요약 — 미래에 관련성 판단에 쓰이므로 구체적으로}
metadata:
  type: {user | feedback | project | reference}
---

{본문 — feedback·project 유형은: 규칙/사실, 그 다음 **Why:** 와 **How to apply:** 줄로 구성}
{관련 메모리는 [[다른-메모리-슬러그]] 로 링크}
```

**2단계** — `MEMORY.md` 인덱스에 한 줄 포인터를 추가한다.
```
- [제목](파일.md) — 한 줄 요약
```
- `MEMORY.md`는 인덱스이지 메모리 본문이 아니다. 각 줄은 ~150자 이내.
- 메모리 본문을 `MEMORY.md`에 직접 쓰지 않는다.

### 운영 규칙
- 주제별(의미별)로 정리한다. 시간순이 아니다.
- 틀렸거나 오래된 메모리는 갱신하거나 제거한다.
- 중복 메모리를 만들지 않는다 — 먼저 갱신할 기존 메모리가 있는지 확인한다.
- 파일이 명시한 함수·경로·플래그를 권고하기 전에, 그것이 **지금도 존재하는지** 실측한다 (메모리는 작성 시점의 사실).

## 3. 이식 방법 — 다른 PC / 다른 Claude Code

### 핵심 원칙: **지식의 원천은 저장소(repo) 안에 둔다.**
그래야 `git clone`만으로 방법론과 누적 지식이 함께 따라간다.

### 무엇이 git으로 따라가고, 무엇이 안 따라가나
| 위치 | 따라가나 | 항목 |
|---|---|---|
| **저장소 안** | git으로 이동 | `CLAUDE.md`, 방법론 .md 세트, `BACKLOG.md`, `TASK_HISTORY.md`, `POLICIES.md` |
| **저장소 안 `memory/` 폴더** | git으로 이동 | 메모리 파일 + `MEMORY.md` 인덱스 |
| **홈 폴더** (`~/.claude/...`) | 이동 안 함 | 머신별 자동 메모리 경로, 전역 설정 |

### 새 PC / 새 인스턴스 설정 절차
1. 저장소를 `git clone` 한다 — `CLAUDE.md` + 방법론 .md + 과업/정책 + `memory/` 가 함께 온다.
2. 해당 폴더에서 Claude Code를 켠다 — 프로젝트 `CLAUDE.md`를 자동으로 읽어 같은 방법론이 적용된다.
3. (선택) 머신 전역 설정이 필요하면 그 머신의 `~/.claude/CLAUDE.md`에 개인 선호를 둔다. 단, **이식에 필요한 것은 모두 저장소 안에 두어** 전역 설정 의존을 없앤다.
