# 개발 코드 규칙 — 근본 철학 + 3대 규칙

## 근본 철학

> **"처음 개발한 코드가 끝까지 쓰이는 경우는 제한적이다. 그래서 지속 가능하고 보완 개발이 가능하도록, 항상 컴팩트(compact)한 코드 작성을 통해 안정성·효율성·지속성·최적화를 꾀한다."**

이 철학이 아래 3대 규칙의 상위 원칙이다. 규칙이 애매한 순간에는 이 철학으로 돌아가 판단한다.

### 네 기둥의 우선순위
| 순위 | 기둥 | 의미 | 주의점 |
|---|---|---|---|
| 1 | **안정성** | 기존 정상 동작을 깨뜨리지 않는다. 변경의 영향 범위를 먼저 본다. | 검증되지 않은 "더 나은 방법"으로 안정된 코드를 섣불리 바꾸지 않는다. |
| 2 | **지속성** | 6개월·1년 뒤에 봐도 의도가 명확하도록 쓴다. | 현 시점만 편한 코드는 지속성의 적이다. |
| 3 | **효율성** | 자원(메모리·쿼리·네트워크)을 낭비하지 않는다. | 효율성을 과하게 추구해 가독성을 희생시키지 않는다. |
| 4 | **최적화** | 필요할 때, 측정한 근거가 있을 때만 한다. | "섣부른 최적화는 모든 악의 근원" (Knuth). 가장 후순위. |

### "컴팩트"의 정의 — 오해 방지
컴팩트는 **짧은 코드 = 좋은 코드**가 아니다.
- 군더더기 없음: 불필요한 중복·죽은 코드·의미 없는 추상화 제거.
- 의도 명확: 한 함수는 한 가지 일. 이름만으로 의도 파악 가능.
- 응집도 높음: 관련 코드는 모으고, 다른 맥락은 분리.
- **아님**: 한 줄로 욱여넣은 "영리한" 코드, 축약어 남용, 매직 넘버.

## 3대 규칙

### 규칙 1 — 파일당 800줄 제한
한 파일의 코드는 800줄을 넘지 않는다.

- **이유**: 파일이 비대해지면 수정 시 영향 범위 파악이 어렵고, 사람·AI 모두 전체 맥락을 놓친다.
- **800줄을 넘으면 리팩토링한다.** 단, 무작정 쪼개지 않고 "가장 안정적이고 효과적인 방식"으로 분리한다 (책임 단위·도메인 경계·재사용성 기준).
- **분리 원칙**:
  - 같은 책임이면 억지로 쪼개지 않는다 (응집도 유지).
  - 다른 책임이 섞여 있으면 분리한다 (단일 책임).
  - 공용 로직은 util/helper로, 도메인 로직은 도메인 모듈로.
- **사전 경고 의무**: 작업으로 파일이 800줄을 넘게 되면, 작업 시작 전 사용자에게 알리고 분리 시점을 협의한다. 승인 없이 무단 리팩토링하지 않는다.

### 규칙 2 — 지속 개발 가능한 구조
운영과 보완 개발이 계속 가능하도록 작성한다.

- 함수·변수명은 의도가 드러나게 (예: `processData` → `sendCampaignEmails`).
- 한 함수는 하나의 일만. 20~40줄 이내 권장.
- 하드코딩 금지 — 설정값은 `.env`나 DB `settings` 테이블로 분리.
- 중복 코드는 추출하되, **3회 이상 반복될 때까지는** 성급하게 추상화하지 않는다 (Rule of Three).
- 주석은 **왜(Why)** 를 남긴다. 무엇(What)은 코드가 말하게 한다.

### 규칙 3 — 보안을 고려한 코드
모든 코드는 작성 시점부터 보안을 고려한다.

**기본 체크리스트** (매 작업마다 자체 검토):
- 사용자 입력 검증 (SQL Injection, XSS, 경로 조작)
- 인증·인가 체크 누락 없는지 (특히 API 라우트)
- 비밀값 하드코딩 금지 (.env 또는 암호화 저장소)
- 에러 메시지에 내부 정보 노출 금지 (스택 트레이스, DB 쿼리)
- 로그에 민감정보 마스킹 (이메일·전화번호·비밀번호)
- HTTPS/TLS 설정 유지 (개발 편의로 `rejectUnauthorized: false` 상시 적용 금지)
- 의존성 취약점 점검 (`npm audit` 주기적 실행)

### 적용 방식
- 코드 작성/수정 **전**에 3대 규칙 위반 여부를 자체 점검한다.
- 위반 가능성이 있으면 작업 시작 전 사용자에게 먼저 알린다.
- 이미 규칙을 위반한 기존 코드를 발견하면 사용자에게 보고한다. 단, 승인 없이 무단 리팩토링하지 않는다.
