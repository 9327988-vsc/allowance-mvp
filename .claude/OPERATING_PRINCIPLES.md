# 운영 원칙 — 작업·검증·표현 의무

모든 작업에 의무 적용하는 본 원칙. 어시스턴트는 코드 작성·검증·보고의 각 국면에서 아래를 지킨다.
(번호는 원 체계를 유지한다. 빠진 번호는 향후 등록 가용.)

---

## 실측 의무
**추정·가정을 피하고 실측한다.** 권고·결정을 내리기 전에 실제 코드·DB·git 이력을 점검한다.

- `grep`·`git blame`·DB 통계·코드 본문 read로 사실을 확인한 뒤 진행한다.
- BACKLOG 본문·메모리 본문에만 의존하지 않는다 (stale 위험).
- 추정한 수치를 단정하지 않는다 — 실측한 수치를 쓴다.

## 검증 의무
**코드 수정 후 grep·git diff·실측으로 검증한 뒤 commit한다.**

- 단일 grep 카운트만으로 "통과" 판정하지 않는다.
- 변경 영역의 라인 범위를 한정해 grep한다 (편집 영역 ±20라인 → 편집 영역 점검과 통합).
- 본 변경 영역과 무관한 일치 라인을 분리해 점검한다.
- 다중 IIFE·구조체 영역의 닫음 위치를 grep할 때는 인접 라인을 함께 출력해 시각 점검한다.

## 표현 점검 의무
**보편적이고 읽기 쉬운 어휘를 쓴다.** 비표준 표현·자기만의 조어를 피한다.

- 정형화된 슬롯에 무의식적으로 같은 단어를 반복 부착하지 않는다.
- 답변 작성 후 자기 점검한다 (특정 표현을 남발하지 않았는지).
- 전문 용어를 쓸 때는 짧게 풀어 설명하거나 예시를 곁들인다.

## URL/API 호환성 점검 의무
**URL·API를 변경할 때 옛 데이터·옛 클라이언트·옛 발송 건의 회귀를 점검한다.** 즉시 폐기보다 단계적 deprecation을 우선한다.

**점검 절차 4단계**:
1. 옛 데이터 잔존 점검 (grep + git blame + DB query)
2. 옛 클라이언트·옛 발송 영향 범위 추정
3. 폐기 분기점 결정 — (가)즉시 / (나)호환 유지+deprecated / (다)후속 트랙 / (라)영구
4. 폐기 일정을 별도 과업으로 등록

**기본 권고**: (나) 호환 유지 + deprecated 주석 + 폐기 일정 별도 등록.

## 편집 영역 ±20라인 점검 의무
**코드 편집 후 grep 검증 시 편집 영역 인접 ±20라인을 함께 출력해 시각 점검한다.**

- 다중 IIFE·구조체·중첩 영역에서 grep 결과 라인이 본 편집 영역과 무관할 수 있다.
- 본 영역의 닫음 위치를 누락하면 빌드 오류·런타임 차단으로 이어진다.

**점검 절차**:
1. 편집 영역 라인 번호 확정
2. 점검 대상 식별자·구조로 grep 패턴 작성
3. 편집 영역 ±20라인 범위를 본문 출력
4. 매칭 라인 인접을 출력해 본 영역 vs 무관 영역 구분
5. 본 영역 닫음 위치를 확인한 뒤 commit

## 동일 파일 연속 commit 누적 점검 의무
**동일 파일에 연속으로 3회 이상 commit하면 누적 영역을 점검한다.**

- 단일 commit 검증·단일 편집 영역 점검을 통과해도, 다중 commit 누적 시 중간 영역 결함이 후속 commit에 가려져 표면화가 늦어진다.

**점검 절차**:
1. `git log --oneline <파일경로> | head -5`
2. `git diff <첫 commit>..HEAD <파일경로>` 로 누적 변경 grep
3. 누적 변경 영역의 닫음 위치를 ±20라인 점검
4. 다음 작업 진입 직전 누적 점검 통과

## 응답 자기 점검 의무
**응답을 마치기 직전, 표현·검증 자기 점검을 수행한다.**

- 비표준 표현·미검증 단정이 응답에 남아 있는지 자기 점검한다.
- 의무 영역: 본문 답변, 결과 보고, 자기 점검 결과 보고, 권고 영역. 어느 한 영역도 면제하지 않는다.
- 자기 점검·메타 분석·결함 보고 답변도 본 원칙의 면제 대상이 아니다.
