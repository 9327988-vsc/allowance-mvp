# Karpathy 4원칙 — 최상위 개발 원칙

> 출처: https://github.com/forrestchang/andrej-karpathy-skills/blob/main/CLAUDE.md
> "These guidelines bias toward caution over speed." — 속도보다 신중함을 우선한다.

모든 작업의 최상위 원칙. 다른 규칙과 충돌하면 이 4원칙으로 돌아가 판단한다.

## 1. Think Before Coding — 코딩 전에 생각한다
- 가정을 명시적으로 표현한다. 불확실하면 묻는다.
- 다수 해석이 가능하면 **모두 제시**한다 (조용히 하나를 택하지 않는다).
- 더 단순한 대안이 있으면 말한다. 합당한 경우 반대 의견을 낸다.
- 불명확하면 멈추고 묻는다.

## 2. Simplicity First — 단순함을 우선한다
- 요청한 것만 만든다. 추상화·유연성·설정 가능성은 요청이 있을 때만.
- 일어날 수 없는 시나리오에 에러 처리를 넣지 않는다.
- **"200줄을 50줄로 줄일 수 있으면 재작성한다."**
- 자기 검증: *"시니어 엔지니어가 보면 과하게 복잡하다고 할까?"*

### 2-A. 단순 대안 검토 의무
비표준 패턴(IIFE·중첩 클로저·복잡한 구조체 등)을 도입하기 직전에 단순 대안을 검토한다.

**점검 절차 4단계**:
1. **자문**: "이 패턴이 200줄→50줄 자문에서 어디에 위치하는가?"를 명시한다.
2. **후보 명시**: 단순 대안 후보 2~3개를 적는다 (예: 상단 const 추출·일반 함수·조건부 렌더링).
3. **근거 작성**: 채택안 vs 미채택 후보를 가독성·유지보수·표준 정합 3축으로 비교한다.
4. **기록**: 미채택 후보와 채택 근거를 commit 메시지 또는 메모리에 남긴다.

**근거**: 비표준 패턴을 도입하면 작성 시점의 단순성 자문이 없을 때 후속 검증·편집·누적 commit 영역에서 결함이 잠복할 위험이 있다. 사전 검토(2-A)가 통과되면 사후 검증 부담이 줄어든다.

## 3. Surgical Changes — 외과적 변경만 한다
- 인접 코드·주석·포맷을 임의로 "개선"하지 않는다.
- 망가지지 않은 것을 리팩토링하지 않는다.
- 기존 스타일을 따른다 (개인 취향과 달라도).
- **모든 변경 라인이 사용자 요청에 직접 추적되어야** 한다.
- 변경으로 인해 고아가 된 import·변수만 제거한다. 기존 dead code를 임의로 삭제하지 않는다.

## 4. Goal-Driven Execution — 검증 가능한 성공 기준
- "검증 추가" → "잘못된 입력으로 테스트 → 통과시키기"처럼 **검증 단계를 명시**한다.
- 다단계 작업은 짧은 계획 + 각 단계 검증으로 나눈다.

## 작동 신호
- diff에 불필요한 변경이 줄어든다.
- 과복잡으로 인한 재작성이 줄어든다.
- 실수 후가 아니라 구현 전에 명확화 질문이 나온다.
